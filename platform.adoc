This guide demonstrates how to create CloudBees Unify workflows and populate analytics dashboards using a **ready-to-use demo repository** with native CloudBees actions.

*Demo Repository*: https://github.com/xgalanxhi/sdlc-metrics-unify

*Workflow Format*: `.cloudbees/workflows/` using CloudBees DSL (`apiVersion: automation.cloudbees.io/v1alpha1`)

*Actions Used*: Native `cloudbees-io/*` actions

=== What This Journey Provides

This journey provides a **pre-configured demo repository** that you can fork and connect to CloudBees Unify to immediately start populating analytics dashboards.

The repository includes:

* *Complete CI/CD Workflow*: Pre-built workflow with test, lint, build, and deployment jobs
* *Analytics Actions*: All CloudBees actions needed for metrics (build artifacts, deployments, test results, evidence)
* *Multi-Environment Pipeline*: Development, staging, and production deployments
* *Production Failure Simulation*: Workflow input to test deployment failure scenarios
* *Ready to Run*: Just fork, connect, and execute

By the end of this guide, you will:

* Fork the demo repository to your GitHub account
* Create a component in CloudBees Unify connected to your forked repository
* Execute the pre-configured workflow to populate analytics dashboards
* Understand which CloudBees actions contribute to each metrics tab
* View and interpret metrics across five analytics dashboards
* Use this as a reference for implementing workflows in your own projects

*Target Audience*: Teams adopting CloudBees Unify for CI/CD and analytics

*Estimated Time*: 30-45 minutes

```
sdlc-metrics-unify/
├── .cloudbees/
```

[[component-setup]]
=== Chapter 1: Component Creation and Workflow Setup

This chapter covers creating a component in CloudBees Unify connected to your forked demo repository.

**CloudBees Documentation**: https://docs.cloudbees.com/docs/cloudbees-unify/latest/organizations-components/manage-components

==== Understanding Components

Components in CloudBees Unify are the foundation for running workflows and collecting analytics. Each component:

* Connects to a source code repository
* Executes workflows defined in `.cloudbees/workflows/`
* Collects metrics and analytics data
* Displays runs, artifacts, and test results

==== Component Creation Steps

===== Step 1: Ensure Repository is Forked

Before creating a component, verify you have forked the demo repository (see <<getting-started>>):

* Your forked repository should be at: `https://github.com/<YOUR_USERNAME>/sdlc-metrics-unify`
* The repository should contain `.cloudbees/workflows/ci-cd.yaml`

===== Step 2: Create a Component

. Log in to https://cloudbees.io
. Navigate to your organization
. Click *Components* in the left navigation
. Click *Create Component* button

===== Step 3: Configure Component Details

*Component Configuration*:

. **Name**: Enter `sdlc-metrics-unify` (or your preferred name)
. **Repository URL**: Enter your forked repository URL: `https://github.com/<YOUR_USERNAME>/sdlc-metrics-unify`
. **SCM Provider**: Select your SCM provider
. **Authentication**: Configure repository access (see tip below)
. Click *Create*

[TIP]
====
*SCM Integration Setup*:

If you haven't connected your SCM provider yet:

. Navigate to *Organization Settings* > *Integrations*
. Select *Source Control Management*
. Add your SCM provider (GitHub, GitLab, Bitbucket)
. Follow the OAuth/App installation flow
. Grant necessary repository permissions

**Documentation**: https://docs.cloudbees.com/docs/cloudbees-unify/latest/integrations/scm-integration
====

===== Step 4: Verify Workflow Files Exist

The forked repository already contains a complete CloudBees workflow. Verify it exists:

**Workflow Location**: `.cloudbees/workflows/ci-cd.yaml`

The workflow includes:

* **Test Job**: Runs Jest tests and publishes results to Test Insights
* **Lint Job**: Runs ESLint code quality checks
* **Security Scan Job**: Placeholder for security scanning
* **Build Job**: Builds the application, registers build artifacts, adds labels, and publishes build evidence
* **Deploy Jobs**: Three deployment jobs (dev, staging, production) with deployment artifact registration and evidence

===== Step 5: Trigger Workflow Execution

Since the forked repository already contains the workflow, you can trigger it manually:

. Navigate to your component in CloudBees Unify
. Click the *Runs* tab
. Click *Run Workflow* button
. Select the `CI/CD Pipeline` workflow
. Configure workflow inputs:
  * **prod_deployment_status**: Choose `success` or `failure` to simulate deployment outcomes
. Click *Run*

===== Step 6: Verify Workflow Execution

. Navigate to your component in CloudBees Unify
. Click *Runs* tab
. Verify your workflow appears and executes
. Check the run details for success/failure status

[NOTE]
====
*Expected Result*:

After pushing the workflow file:

* Workflow appears in the component's Runs tab
* First run triggers automatically on push to main branch
* Run details show step execution and logs
* Component dashboard updates with run statistics

If the workflow doesn't trigger:

* Verify the file is in `.cloudbees/workflows/` (not `.github/workflows/`)
* Check the workflow YAML syntax is valid
* Ensure the component is connected to the correct repository
* Review component settings for branch configuration
====

==== Build Artifact Registration in Demo Workflow

The demo workflow already includes build artifact registration using the `cloudbees-io/register-build-artifact@v2` action.

**Action Documentation**: https://docs.cloudbees.com/docs/cloudbees-unify/latest/artifact-management/register-build-artifact

**From `.cloudbees/workflows/ci-cd.yaml` in the demo repository**:

[source,yaml]
----
  build:
    needs:
      - test
      - lint
      - security-scan
    outputs:
      version: ${{ steps.version.outputs.version }}
      artifact_id: ${{ steps.register-artifact.outputs.artifact-id }}
    steps:
      - name: Checkout code
        uses: cloudbees-io/checkout@v2
        with:
          fetch-depth: 0

      - name: Generate version and build
        id: version
        uses: docker://node:18
        shell: sh
        run: |
          git config --global --add safe.directory /cloudbees/workspace
          COMMIT_COUNT=$(git rev-list --count HEAD)
          SHORT_SHA=$(git rev-parse --short HEAD)
          VERSION="1.0.${COMMIT_COUNT}-${SHORT_SHA}"
          printf "%s" "$VERSION" > $CLOUDBEES_OUTPUTS/version
          echo "Generated version: $VERSION"

          npm ci
          npm ci --production
          mkdir -p dist
          cp -r src/ dist/
          cp -r node_modules/ dist/
          cp package.json dist/

          cd dist
          tar -czf ../${{ env.APP_NAME }}-${VERSION}.tar.gz .

      - name: Register Build Artifact in CloudBees Unify
        id: register-artifact
        uses: cloudbees-io/register-build-artifact@v2
        with:
          name: ${{ cloudbees.scm.repositoryUrl }}
          url: https://artifacts.example.com/${{ env.APP_NAME }}/${{ steps.version.outputs.version }}
          version: ${{ steps.version.outputs.version }}
          type: Binary

      - name: Add Labels to Artifact
        uses: cloudbees-io/label-artifact-version@v1
        with:
          artifact-id: ${{ steps.register-artifact.outputs.artifact-id }}
          labels: ${{ cloudbees.scm.branch }},nodejs,express,automated

      - name: Publish Build Evidence
        uses: cloudbees-io/publish-evidence-item@v1
        with:
          content: |-
            # Build Evidence Report

            ## Artifact Details
            - **Repository**: ${{ cloudbees.scm.repositoryUrl }}
            - **Version**: ${{ steps.version.outputs.version }}
            - **Artifact ID**: ${{ steps.register-artifact.outputs.artifact-id }}
            - **Type**: NPM Package
            - **Branch**: ${{ cloudbees.scm.branch }}
----

**View the complete workflow**: https://github.com/xgalanxhi/sdlc-metrics-unify/blob/main/.cloudbees/workflows/ci-cd.yaml

===== Artifact Registration Parameters

[cols="30%,50%,20%",options="header"]
|===
| Parameter
| Description
| Required

| `name`
| Name of the artifact
| Yes

| `version`
| Version identifier (typically commit SHA)
| Yes

| `url`
| URL where artifact is stored
| Yes

| `digest`
| Hash/checksum for artifact verification
| No

| `labels`
| Comma-separated list of labels
| No
|===

[TIP]
====
*Best Practices for Artifact Registration*:

* Use commit SHA for version: `${{ cloudbees.scm.sha }}`
* Use descriptive artifact names: `my-app-frontend`, `api-service`
* Register artifacts only after successful build
* Include digest when available for verification
* Use labels for categorization: `production`, `staging`, `feature`
====

===== Full Example with Multiple Artifact Types

[source,yaml]
----
apiVersion: automation.cloudbees.io/v1alpha1
kind: workflow
name: Multi-Artifact Build

on:
  push:
    branches:
      - main

jobs:
  build-frontend:
    steps:
      - name: Checkout
        uses: cloudbees-io/checkout@v1

      - name: Build frontend
        uses: docker://node:20-alpine
        run: |
          npm install
          npm run build

      - name: Register frontend artifact
        uses: cloudbees-io/register-build-artifact@v2
        with:
          name: my-app-frontend
          version: ${{ cloudbees.scm.sha }}
          url: s3://my-bucket/frontend-${{ cloudbees.scm.sha }}.tar.gz
          labels: frontend,production

  build-backend:
    steps:
      - name: Checkout
        uses: cloudbees-io/checkout@v1

      - name: Build backend
        uses: docker://maven:3.9-eclipse-temurin-17
        run: |
          mvn clean package

      - name: Register backend artifact
        uses: cloudbees-io/register-build-artifact@v2
        with:
          name: my-app-backend
          version: ${{ cloudbees.scm.sha }}
          url: docker.io/myorg/backend:${{ cloudbees.scm.sha }}
          digest: sha256:abcdef1234567890
          labels: backend,java,production
----

==== Example: Gitleaks Secret Scanning

**Action Documentation**: https://docs.cloudbees.com/docs/cloudbees-unify/latest/secret-scanners/gitleaks-plugin

Gitleaks scans your repository for exposed secrets and credentials.

===== Basic Gitleaks Workflow

[source,yaml]
----
apiVersion: automation.cloudbees.io/v1alpha1
kind: workflow
name: Security Scanning

on:
  push:
    branches:
      - main
  pull_request:
    branches:
      - main

jobs:
  secret-scan:
    steps:
      - name: Checkout code
        uses: cloudbees-io/checkout@v1

      - name: Scan for secrets
        uses: cloudbees-io/gitleaks-plugin@v1
----

===== Gitleaks with Quality Gates

The Gitleaks action provides output variables for quality gate decisions:

[source,yaml]
----
apiVersion: automation.cloudbees.io/v1alpha1
kind: workflow
name: Security with Quality Gates

on:
  push:
    branches:
      - main

jobs:
  security-scan:
    steps:
      - name: Checkout
        uses: cloudbees-io/checkout@v1

      - id: gitleaks
        name: Scan for secrets
        uses: cloudbees-io/gitleaks-plugin@v1

      - name: Check scan results
        uses: docker://alpine:latest
        run: |
          echo "Critical count: ${{ steps.gitleaks.outputs.critical-count }}"
          echo "High count: ${{ steps.gitleaks.outputs.high-count }}"
          if [ "${{ steps.gitleaks.outputs.critical-count }}" -gt "0" ]; then
            echo "Critical vulnerabilities found!"
            exit 1
          fi
----

==== Configuration

Test Insights requires the `cloudbees-io/publish-test-results@v2` action to upload test results to CloudBees Unify.

===== Basic Test Workflow

[source,yaml]
----
apiVersion: automation.cloudbees.io/v1alpha1
kind: workflow
name: Test Pipeline

on:
  push:
    branches:
      - main

jobs:
  test:
    steps:
      - name: Checkout
        uses: cloudbees-io/checkout@v1

      - name: Run tests
        uses: docker://maven:3.9-eclipse-temurin-17
        run: |
          mvn test

      - name: Publish test results
        uses: cloudbees-io/publish-test-results@v2
        with:
          test-type: junit
          results-path: target/surefire-reports/*.xml
----

===== Test Results Parameters

| Description

* **JUnit** - `test-type: junit` or `test-type: JUNIT`

[[dora-metrics]]
=== Chapter 5: DORA Metrics

DORA (DevOps Research and Assessment) metrics are industry-standard measures of software delivery performance, including Deployment Frequency, Lead Time for Changes, Mean Time to Recovery, and Change Failure Rate.

**CloudBees Documentation**: https://docs.cloudbees.com/docs/cloudbees-unify/latest/analytics/dora-metrics

**Action Documentation**: https://docs.cloudbees.com/docs/cloudbees-unify/latest/artifact-management/register-deployed-artifact

==== What This Dashboard Shows

* **Deployment Frequency**: How often code is deployed to production
* **Lead Time for Changes**: Time from commit to production deployment
* **Mean Time to Recovery (MTTR)**: Time to recover from failures
* **Change Failure Rate**: Percentage of deployments causing failures

==== Configuration

DORA Metrics are populated using the `cloudbees-io/register-deployed-artifact@v2` action to track deployments.

==== Registering Deployments

To track deployments for DORA metrics, use `register-deployed-artifact` in your deployment workflow:

===== Basic Deployment Registration

[source,yaml]
----
apiVersion: automation.cloudbees.io/v1alpha1
kind: workflow
name: Deploy to Production

on:
  workflow_dispatch:

jobs:
  deploy:
    steps:
      - name: Checkout
        uses: cloudbees-io/checkout@v1

      - name: Deploy application
        uses: docker://kubectl:latest
        run: |
          kubectl apply -f k8s/deployment.yaml

      - name: Register deployment
        uses: cloudbees-io/register-deployed-artifact@v2
        with:
          name: my-app
          version: ${{ cloudbees.scm.sha }}
          url: docker.io/myorg/myapp:${{ cloudbees.scm.sha }}
          environment: production
----

===== Deployment Parameters

[cols="30%,50%,20%",options="header"]
|===
| Parameter
| Description
| Required

| `name`
| Artifact name
| Yes

| `version`
| Version being deployed (typically commit SHA)
| Yes

| `url`
| Location of the deployed artifact
| Yes

| `environment`
| Target environment (production, staging, etc.)
| Yes

| `status`
| Deployment status (success, failure)
| No
|===

==== Complete Build and Deploy Example

This example shows the full pipeline from build to deployment with DORA tracking:

      - name: Register production deployment
        uses: cloudbees-io/register-deployed-artifact@v2
        with:
          name: my-app
          version: ${{ cloudbees.scm.sha }}
          url: docker.io/myorg/myapp:${{ cloudbees.scm.sha }}
          environment: production

=== Chapter 6: Flow Metrics

Flow Metrics track work item efficiency, lead time, cycle time, and bottleneck identification throughout the software delivery lifecycle.

==== What This Dashboard Shows

* Lead time from issue creation to deployment
* Cycle time for work items
* Work in progress (WIP) trends
* Bottleneck identification
* Flow efficiency metrics
* Workload distribution and capacity planning

==== Configuration Requirements

[IMPORTANT]
.JIRA INTEGRATION REQUIRED
====
**Flow Metrics require Jira integration to light up.**

*Requirements*:

* Jira instance connected to CloudBees Unify
* Flow items mapping configured in Analytics Configuration
* Work items (issues, stories) linked to commits/PRs via ticket IDs

*Important*: Webhooks are NOT required. Flow Metrics use scheduled batch polling and event triggers.
====

==== How Flow Metrics Data Synchronization Works

CloudBees Unify collects Flow Metrics data through scheduled polling and automatic event triggers, eliminating the need for webhook configuration.

===== Default Polling Intervals

[cols="40%,60%",options="header"]
|===
| Data Type
| Polling Frequency

| Projects
| Every 24 hours

| Issues
| Every 3 hours

| Issue Type Mappings
| Every 24 hours
|===

===== Immediate Processing Triggers

Data synchronization is triggered immediately when:

* **Endpoint CREATE/UPDATE Events**: When a Jira endpoint is created or updated in CloudBees Unify
* **Analytics Config Updates**: When Flow item mappings are modified in Analytics Configuration

[TIP]
====
After configuring or modifying Jira integration or Flow item mappings, data will begin syncing immediately. For routine updates, polling occurs automatically at the intervals above.
====

==== Configuration Steps

===== Step 1: Configure Jira Integration

. Log in to CloudBees Unify
. Navigate to *Integrations*
. Add or configure your Jira integration
. Provide Jira instance URL and authentication credentials
. Verify connection is successful

[NOTE]
====
This step assumes you have administrative access to configure integrations in CloudBees Unify. Contact your CloudBees administrator if you need assistance with Jira integration setup.
====

===== Step 2: Configure Flow Items Mapping

Flow items mapping tells CloudBees Unify which Jira issue types correspond to which flow categories.

. Navigate to *Analytics* > *Analytics Configuration*
. Locate the *Flow Items Mapping* section
. Map your Jira issue types to flow categories:

[cols="30%,70%",options="header"]
|===
| Flow Category
| Description

| Feature
| User stories, new features, enhancements

| Defect
| Bugs, issues, defects

| Risk
| Security vulnerabilities, technical risks

| Technical Debt
| Refactoring, code improvements, tech debt items
|===

*Mapping Options*:

* **By Issue Type**: Select Jira issue types (Story, Bug, Task, etc.)
* **By JQL Query**: Use Jira Query Language for advanced filtering

[source,text]
----
Example JQL Mappings:

Feature: issuetype = Story OR issuetype = Epic
Defect: issuetype = Bug
Risk: labels = "security" OR labels = "risk"
Technical Debt: labels = "tech-debt" OR issuetype = "Technical Debt"
----

[TIP]
====
*Best Practices for Flow Item Mapping*:

* Use consistent issue types across your Jira projects
* Leverage labels for cross-cutting concerns (security, tech-debt)
* Test mappings with JQL queries in Jira first
* Ensure all active issue types are mapped to at least one category
====

===== Step 3: Link Commits to Jira Issues

To enable tracking from code commits to Jira issues, include Jira ticket IDs in commit messages.

*Commit Message Patterns*:

[source,text]
----
# Bracket notation
feat: add user authentication [PROJ-123]

# Parentheses notation
fix: resolve login timeout issue (PROJ-456)

# Prefix notation
PROJ-789: implement dashboard widgets

# Multiple tickets
PROJ-123 PROJ-456: refactor authentication module
----

CloudBees Unify automatically detects Jira issue keys (e.g., `PROJ-123`) in commit messages and links them to work items.

[IMPORTANT]
====
*Ticket ID Format*:

Jira ticket IDs must follow the standard format: `PROJECT-NUMBER`

* Valid: `PROJ-123`, `ABC-456`, `PLATFORM-789`
* Invalid: `proj-123` (lowercase), `123-PROJ` (reversed)
====

   * Refresh Flow Metrics dashboard
   * Ticket should reflect "Done" status
   * Cycle time and lead time metrics should be calculated

[NOTE]
====
*Data Refresh Timing*:

* **Immediate**: If you modify Analytics Configuration after committing
* **Within 3 hours**: For issue status updates (next polling cycle)

[[troubleshooting]]
== Troubleshooting

This section provides solutions to common issues encountered when implementing CloudBees Unify workflows and analytics.

=== Component Creation Issues

**Issue**: Component creation fails or repository connection doesn't work

*Solution*:

* Verify SCM integration is properly configured in Organization Settings
* Check that you have necessary permissions in the source repository
* Ensure repository URL is correct and accessible
* For private repositories, verify authentication credentials are valid

**Issue**: Workflows don't appear after creating component

*Solution*:

* Confirm workflow files are in `.cloudbees/workflows/` directory
* Verify YAML syntax is valid using a YAML validator
* Check that `apiVersion: automation.cloudbees.io/v1alpha1` is present
* Push changes to the branch configured in component settings

=== Workflow Execution Issues

**Issue**: Workflow fails with "action not found" error

*Solution*:

* Verify action namespace is `cloudbees-io/*` (not `actions/*` or `cloudbees-io-gha/*`)
* Check action version is correct (e.g., `@v1`, `@v2`)
* Ensure action name is spelled correctly

**Issue**: Context variables are undefined or empty

*Solution*:

* Use CloudBees context variables: `${{ cloudbees.scm.sha }}`, `${{ cloudbees.scm.branch }}`, etc.
* Verify context variable exists in CloudBees DSL documentation
* Check variable is appropriate for the workflow trigger type

=== Analytics Dashboard Issues

**Issue**: Software Delivery Activity dashboard is empty

*Solution*:

* Verify at least one workflow run has completed successfully
* Check `register-build-artifact@v2` action is present in workflow
* Confirm artifact registration step completed without errors
* Allow 5-10 minutes for data to populate after workflow completion

**Issue**: Test Insights shows no data

*Solution*:

* Verify `publish-test-results@v2` action is in workflow
* Confirm test framework outputs JUnit XML format
* Check `results-path` parameter points to correct test result files
* Ensure `test-type` parameter matches your testing framework
* Verify test step runs even when tests fail

**Issue**: Security Insights dashboard empty

*Solution*:

* Confirm security scanning action is present in workflow
* Check security scan step completed successfully
* Review workflow logs for security action errors
* Verify component has access to required security tools
* Allow time for scan results to process and display

**Issue**: DORA Metrics not calculating

*Solution*:

* Verify `register-deployed-artifact@v2` action is used for deployments
* Check that both build and deployment registrations use matching artifact names and versions
* Confirm deployments are registered to production environment
* Ensure sufficient data exists (typically 2-4 weeks for meaningful metrics)

**Issue**: Flow Metrics empty or incomplete

*Solution*:

* Verify Jira integration is configured in CloudBees Unify
* Check Flow items mapping is configured in Analytics Configuration
* Confirm commit messages include Jira ticket IDs (e.g., PROJ-123)
